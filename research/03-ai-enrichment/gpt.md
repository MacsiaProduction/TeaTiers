# Рекомендуемая архитектура

- **Основной (Яндекс Cloud):** YandexGPT через их Foundation Models API – домашняя LLM для русского и англ. (высокое качество и локальный хостинг). Для перевода названий можно дополнительно пробовать Yandex Translate API (дешево, но без пиньиня и с известными ошибками по именам).
- **Резервные (Free-tier):** Groq API – обширный бесплатный слой LLM с приличными лимитами (подойдёт для экспериментов). OpenRouter/Hugging Face – агрегаторы открытых моделей (DeepSeek, Mistral и др.) с ограниченными бесплатными запросами. При необходимости проверки/перевода коротких строк можно использовать Google Translate через HF-интеграцию или DuckDuckGo.
- **Путь верификации:** Wikidata (SPARQL/API) – первичная проверка через фактическую БД с CC0-данными. Если Wikidata не дала результат или дает сомнительно, LLM (выбор модели по свободному доступу) → проверка полей (например, запрос в Вики или поиск имени с помощью Qwant API и Tavily) → публикация с флагом «неверифицировано» и оценкой доверия.

# Схема конвейера

1. **Поиск в Wikidata:** Поиск по имени чая (включая альтернативные метки) через API или SPARQL. Если найдена запись, заполняем поля `ru_name`, `en_name`, `zh_name` (этикетки) и свойства (`type`, `origin`, `oxidation`). Это «верифицированные» данные, ставим высокое `confidence` (напр. 0.9).
2. **LLM (fallback):** Если Wikidata не дала полного результата, формируем запрос LLM. Пример инструкции модели (JSON-формат):
   ```json
   {
     "name_input": "<название чая на одном языке (user input)>",
     "result": {
       "ru_name": "",
       "en_name": "",
       "zh_name": "",
       "pinyin": "",
       "type": "",
       "origin": "",
       "oxidation": "",
       "confidence": 0.0
     }
   }
   ```
   Модель должна заполнить поля и оценить `confidence` (0–1). Например:  
   - `ru_name, en_name, zh_name, pinyin` – названия и транслитерация.  
   - `type` – тип чая (зеленый/черный/улун и т.д.).  
   - `origin` – регион происхождения (страна/место).  
   - `oxidation` – степень ферментации (обычно «низкая», «средняя», «высокая»).  
   - `confidence` – примерная уверенность модели в ответе (можно попросить модель оценить по результатам своего анализа).
3. **Кросс-проверка:** Автоматический поиск по выданному результату (пример: запрос в Qwant API с названием чая + ключевые слова «тип»/«ферментация») или быстрый SPARQL по Wikidata ещё раз. Сравниваем с ответом LLM: совпадения повышают доверие, расхождения – понижают. (Можно также учесть источник: если LLM подтвердила то же, что есть в Википедии/Викиданных, повысить `confidence`.) Итоговая `confidence` может вычисляться как взвешенная метрика (например, 0.5 от совпадений по признакам из Wikidata + 0.5 от уверенности LLM).
4. **Публикация:** Записываем в БД новые поля и `confidence`. Все записи помечаются флагом «unverified» до ручной модерации. Логика дублирования: создаём уникальный ключ по нормализованному названию (lowercase, без пробелов/спецсимволов) + pinyin (slug), чтобы избежать повторной обработки одного и того же чая.

# Защита квот и от злоупотреблений

- **Кэширование:** Результаты разрешения названия (Wikidata/LLM) сохраняются. Повторный запрос по тому же названию выдаёт кешированный ответ (защитит от повторных обращений к LLM).
- **Ограничение запросов:** Внедрить rate-limiting на endpoint: например, не более 5–10 запросов в минуту от одного IP, максимум 100–200 в сутки. Это предотвратит допросы LLM с целью «слить» бесплатный квант или DDOS.
- **Валидация ввода:** Допускать только разумные названия: ограничить длину строки (например, 100 символов), блокировать подозрительные символы/HTML. Проверять, что введён текст (не SQL/JS). Резко снижаем риск инжекций.
- **Защита от Prompt Injection:** Формировать системный промпт так, чтобы пользовательский ввод лишь подставлялся в поле `name_input`, а LLM возвращала строго JSON (например, дополнительно запросить «не выводите ничего кроме JSON с полями»). Парсить ответ строго по ожидаемой схеме (валидатор JSON), игнорировать лишние данные.
- **Бюджетные стоп-лоссы:** Отключать LLM-фоллбек, если достигнут месячный лимит вызовов (по бесплатным сервисам) или использовать альтернативы. Мониторить расходы (особенно Yandex) и настройки cost control в облаке.
- **Аутентификация/MFA для чувствительных запросов:** (опционально) внедрить CAPTCHA или API-ключ для «подозрительных» вызывающих (многочисленные запросы с одного адреса).
- **Логирование:** Журналировать обращения и ответы LLM, чтобы при аномальном трафике можно было оперативно среагировать и заблокировать злоумышленника.

# Источники и неопределённое

- Яндекс Foundations: модели и API (официальная документация Yandex Cloud AI Studio). Бесплатный стартовый грант: $30/4000₽.
- Яндекс Перевод: цены ($4.10/млн. символов). Поддержка китайского (zh) и русского есть, но пиньинь **не** выдаёт.
- Groq API: условия бесплатного тарифа, хранение данных по договору.
- OpenRouter/HuggingFace: общие условия бесплатного доступа (см. сайты OpenRouter, HF Terms).
- Qwant API: неофициальный, но бесплатный «безлимитный» доступ.
- Tavily Search: есть бесплатный план (1000 кредитов/мес.).
- Wikidata: данные CC0, бесплатны и общедоступны.
- **Неподтверждённо:** точный размер «бесплатного» тарифа Yandex Cloud LLM (около 90 дней, но без долгосрочных бесплатных лимитов); детализация хранения выходных данных YandexGPT (официально не оговорена); реальная гео-блокировка Google/Gemini (нет прямой Yandex-страницы, но известно практически); эффективность перевода Yandex для имён (экспертный опыт).
