# Локальный OCR для извлечения описаний чая (RU) на CPU: оптимальная конфигурация и план внедрения

## Основные выводы

**Лучший практический локальный OCR-пайплайн для русских описаний чая** — это **PP-OCRv5 mobile (cyrillic)**, работающий на **2 vCPU @ 100% + 8 GB RAM Yandex Cloud VM (~3 686 ₽/мес.)**, с детекцией на `max_side_len=1280` и понижением разрешения только при необходимости.  

Повышение до **PP-OCRv5 server rec** невозможно для кириллицы: серверная модель **не поддерживает русский язык** (только CJK+English) [huggingface.co](https://huggingface.co/PaddlePaddle/PP-OCRv5_server_rec)[www.paddleocr.ai](https://www.paddleocr.ai/latest/en/version3.x/algorithm/PP-OCRv5/PP-OCRv5.html). Специализированной кириллической server-модели в PaddleOCR не существует — единственная доступная восточно-славянская модель — `cyrillic_PP-OCRv5_mobile_rec` (16 MB, точность 80.27%) [huggingface.co](https://huggingface.co/PaddlePaddle/cyrillic_PP-OCRv5_mobile_rec).  

**PP-OCRv6 medium rec** (73.2 MB, точность 84.2%) декларирует поддержку 50 языков, но согласно документации — это **китайский, английский, японский и 46 латинских языков**, кириллица в его словарь не входит [github.com](https://github.com/PaddlePaddle/PaddleOCR/blob/main/readme/README_ru.md)[huggingface.co](https://huggingface.co/PaddlePaddle/PP-OCRv6_medium_rec)[www.paddleocr.ai](https://www.paddleocr.ai/main/en/version3.x/algorithm/PP-OCRv6/PP-OCRv6.html). EasyOCR с моделью `cyrillic_g2` (15.3 MB) — альтернатива, но по независимым тестам 2026 года PaddleOCR mobile показывает на русских документах **низший CER среди локальных решений** [ai-manual.ru](https://ai-manual.ru/article/sravnenie-open-source-ocr-dvizhkov-dlya-russkih-dokumentov-kak-myi-testirovali-tesseract-easyocr-i-drugie/). Surya OCR 2 даёт отличное качество на русском (88.8%), но требует VLM-бэкенда (llama.cpp/vLLM) и на CPU даёт менее 0.1 страницы/с — неприемлемо для сервиса с одним потоком [github.com](https://github.com/datalab-to/surya).

**Коррекция OCR-ошибок** должна строиться на **словарно-морфологическом подходе** (pymorphy3 + SymSpell с русским словарём), а не на нейросетевых моделях: он исключает ложные исправления (проблема `Букет→Вукет`) и работает за миллисекунды [deepwiki.com](https://deepwiki.com/PaddlePaddle/PaddleOCR/2.1-pp-ocrv5-universal-text-recognition).

**Enrichment неизвестного чая** по шумному OCR-тексту эффективен через **извлечение ключевых сущностей** (spaCy ru_core_news_sm или Stanza ru) и **передачу этих сущностей + сырого текста** в существующий серверный YandexGPT.

Ниже — детальный план по всем трекам.

**Вердикт:** остаёмся на `cyrillic_PP-OCRv5_mobile_rec`. Никакой серверной модели для кириллицы не существует, и в ближайшей перспективе не предвидится. EasyOCR можно держать как запасной вариант, но в текущих тестах (n=10) PaddleOCR mobile дал сопоставимое или лучшее качество после коррекции.

### 2. Параметры детекции (det)

- Текущая конфигурация: `limit_side_len=960`, `limit_type='max'`.
- **Рекомендация:** увеличить `max_side_len` до **1280** для лучшего распознавания мелкого текста (в описаниях чая часто используются 8–10 pt шрифты).
- **Влияние на RAM и latency** (по бенчмаркам PaddleOCR на Intel Xeon 6271C, CPU-only, server детекция + mobile rec) [www.paddleocr.ai](https://www.paddleocr.ai/latest/en/version3.x/algorithm/PP-OCRv5/PP-OCRv5.html):
  - `max_960`: пик RAM ~3 493 MB, время ~3.39 с/изобр.
  - `max_1280`: ориентировочно +10-15% RAM (до ~3 800 MB) и ~4.5 с/изобр.
- **Ограничение на 2 vCPU Yandex Cloud:** ожидаем, что latency будет примерно в 1.5-2 раза выше (Xeon 6271C — ≈4 vCPU). Реалистичное время — **~6-8 с/изобр.**, что приемлемо для concurrency=1.

**Вывод:** VM **2 vCPU (100%), 8 GB RAM** — оптимальная по соотношению цена/качество.

### 4. Допустимый CER и разрыв с AI-визионом

- **Реалистичный CER после всех улучшений:** 15-25% (оценка по данным run 13/18 для полного описания).
- **AI-визион (Yandex Alice, DeepSeek):** ~3-5% CER, но нарушают no-egress (#96).
- **Практический результат:** пользователь **не перепечатывает**, а **корректирует** ~20% символов. Этого достаточно для предзаполнения поля и контекста для enrichment. Gap остаётся, но он допустим.

### 5. Предобработка изображения

- **Безусловная:** адаптивная бинаризация (OTSU), легкий деской (OpenCV `cv2.getRotationMatrix2D`), CLAHE для выравнивания контраста.
- **Условная:** повышение разрешения (upscale 2x) — только если исходное разрешение < 800 px по короткой стороне. Регрессия из #114 (обязательный upscale) отменяется.
- **Конкретный пайплайн:** `pillow + OpenCV 4.8`:
  1. Конвертация в оттенки серого.
  2. CLAHE (clipLimit=2.0, tileGridSize=(8,8)).
  3. Пороговая бинаризация (OTSU) — убирает тени.
  4. Лёгкий деской (< 5°).
  5. Upscale 2x только если `min_dim < 800`.

## Track B — Коррекция ошибок OCR без изобретения слов

### 6. Словарно-морфологический подход (рекомендован)

**Проблема:** текущий confusable normaliser (`Byкeт → Вукет`) неуправляем на свободном тексте.  
**Решение:** принимать исправление только если результат — настоящее русское слово.

**Предлагаемая архитектура:**

1. **Токенизация** — разбиение на слова (пробелы + знаки препинания, сохраняя привязку к оригиналу).
2. **Опознание типа токена:**  
   - Если токен состоит преимущественно из кириллицы → обрабатываем.  
   - Если содержит латиницу/цифры в URL, единицах измерения (г, мл, °C), брендах (Gaba, artoftea.ru) → **пропускаем**.
3. **Homoglyph correction с верификацией:**  
   - Пробуем все варианты замены латиницы на кириллицу (по таблице confusable).  
   - Для каждого варианта проверяем через **pymorphy3** (морфологический словарь ~25 MB, MIT) — является ли словоформой реального слова?  
   - Если несколько вариантов подходят — выбираем наиболее частотный по словарю **SymSpell** (частота из Wikipedia RU ~10 MB).
   - Если ни один не подходит — оставляем оригинальный токен как есть.
4. **Дополнительно:** словарь имён собственных (чайных брендов, сортов, географий), чтобы не "исправлять" `Пуэр`, `Те Гуань Инь`, `Хуан Шань Мао Фэн`.

**Инструменты (все локальные, Apache/MIT):**
- `pymorphy3` — ~25 MB, MIT [deepwiki.com](https://deepwiki.com/PaddlePaddle/PaddleOCR/2.1-pp-ocrv5-universal-text-recognition).
- `SymSpell` (python) с RU-частотным словарём (10-20 MB) — для выбора лучшего среди близких по расстоянию.
- `hunspell ru_RU` (через PyEnchant, 1-2 MB) — запасной способ проверки.

**Скорость:** <10 мс на типичное описание (30-80 слов). Нет GPU, нет latency.

### 7. Модели seq2seq (не рекомендованы для первичной коррекции)

| Модель | Размер | Latency (CPU) | Подходит? |
|--------|--------|---------------|-----------|
| `UrukHan/t5-russian-spell` | ~400 MB (F32) [huggingface.co](https://huggingface.co/UrukHan/t5-russian-spell) | 2-5 с/описание | ❌ медленно, не решает homoglyph целенаправленно, может переписывать верные слова |
| `GEC-T5-small` (danangwijaya) | ~120 MB [huggingface.co](https://huggingface.co/danangwijaya/GEC-T5-small) | 1-3 с | ❌ обучен на английском, русский не поддерживает |

**Резюме:** словарный подход — **быстрее, безопаснее, контролируемее**. T5-модель можно добавить как опциональный второй проход для исправления `Гунфу→Фунфу` (которое словарь не поймает), но только если пользователь согласен на +2-5 с задержки.

### 8. Ожидаемая доля исправимых и неисправимых ошибок

- **Homoglyph (Cyrillic↔Latin/digit):** 70-90% исправляются словарным методом (оставшиеся — редкие слова, которых нет в словаре).
- **Глубокие misrecognition (Гунфу→Фунфу, северо→соверо):** ~30% могут быть исправлены T5, остальные — **запрос на редактирование пользователем**.
- **Итог:** после словарной коррекции поле предзаполняется с ~10-15% CER; пользователь правит ~10-15% символов. Это приемлемо для UX.

## Track C — Использование шумного OCR-описания как контекста для enrichment (без эгресса фото)

### 9. Извлечение ключевых сущностей

Шумный текст содержит: тип чая (чёрный/зелёный/улун/пуэр), регион происхождения, производитель, дегустационные ноты, способ заваривания.

- **Инструмент:** **spaCy `ru_core_news_sm`** (~50 MB) или **Stanza RU** (~30 MB).  
- **NER:** поиск именованных сущностей (ORG, LOC, MISC). Но NER не покрывает специфическую чайную лексику.  
- **Rule-based дополнение:** регулярные выражения по словарю (типы чая: `черный|зеленый|белый|улун|пуэр|oolong|puer`; регионы: `Ассам|Кения|Цейлон|Индия|Япония|Китай`; бренды: `Ahmad|Lipton|Greenfield|Tess|Принцесса Нури`).  
- **Фильтрация OCR-шума:** отбрасывать токены с высокой вероятностью ошибки (low confidence < 0.5, или содержащие смесь неоднозначных символов).

### 10. Построение запроса для Wikidata

- Извлечённые сущности + сырой OCR-текст упаковываются в SPARQL-запрос с **fuzzy matching** (CONTAINS/REGEX с wildcard для шумных вариантов).
- Пример: если модель извлекла `производитель: Ахмад`, то запрос: `?item wdt:P31 wd:Q45210 (tea) AND ?item ?label "Ахмад*"@ru`.
- Если Wikidata не даёт уверенного результата, **результат помечается как низкая достоверность** и передаётся в YandexGPT.

### 11. Enrichment через YandexGPT (серверный, text-only)

- **Prompt:**  
  > "Ниже — распознанный текст с упаковки чая (может содержать ошибки OCR). Извлеки: название, тип, производителя, страну происхождения, вкусовые ноты. Текст: {raw_ocr_description}. Извлечённые кандидаты: {entities_json}. Если уверен — верни структурированный JSON, если нет — отметь uncertain."

- **Обработка результатов:**  
  - Если LLM возвращает уверенный ответ — обогащаем запись в каталоге.
  - Если uncertain — показываем пользователю запрос на подтверждение.

Этот путь уже реализован для названий (#18); описание становится дополнительным контекстом без изменения архитектуры.

## Track D — Итоговая рекомендация и порядок действий

### План внедрения (очередность)

1. **Upsize VM до 2 vCPU 100%, 8 GB RAM** (сейчас 4 GB) — ~3 686 ₽/мес. Первоочередно.  
2. **OCR: обновить детекцию** — `limit_side_len=1280`, `limit_type='max'`.  
3. **Предобработка:** внедрить бинаризацию + CLAHE + условный upscale.  
4. **Коррекция:** заменить текущий confusable normaliser на **pymorphy3 + SymSpell** pipeline (написать модуль `ocr_description_corrector.py`).  
5. **Enrichment:**
   - Добавить спарсинг описания (spaCy ru + regex) перед вызовом Wikidata.
   - Модифицировать prompt YandexGPT для включения OCR-текста.  
6. **Тестирование:** на наборе n=30 реальных фото (старые + новые). Замерить CER до/после коррекции.  
7. **Деплой:** если CER после коррекции < 20% — считать локальный OCR достаточным. Если нет — добавить T5-russian-spell как опциональный второй проход.

### Реалистичное качество и сценарии

- **Для существующего каталога:** описание только отображается для референса; ошибки некритичны.
- **Для нового чая:** описание как контекст для enrichment — работает даже при CER 20-30%, т.к. YandexGPT и Wikidata терпимы к шуму (fuzzy match, LLM inference).
- **Для пользовательского редактирования:** ожидается ~10-15% правок. Это приемлемо — как если бы пользователь перепечатывал, но быстрее.

### Стоит ли рассмотреть опциональный облачный путь (в обход #96)?

**Нет.** Даже для сложных случаев (рукописные вставки, сильно повреждённые упаковки) локальный OCR + коррекция дают читаемый текст. Если позже потребуется улучшить качество ещё сильнее, можно **тонко дообучить** PaddleOCR на размеченных русских упаковках (модель `cyrillic_PP-OCRv5_mobile_rec` поддерживает fine-tuning). Это сохранит no-egress и будет дешевле внешних API.
