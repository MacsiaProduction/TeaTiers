# Achieving High-Fidelity Local OCR for Russian Tea Descriptions: An Integrated Strategy for VM Scaling, Safe Correction, and Enrichment Context Generation

## Optimizing the Local CPU OCR Pipeline with Server-Grade Models

The foundational step in achieving the project's goal of extracting clean, faithful full-text Russian descriptions lies in upgrading the Optical Character Recognition (OCR) engine itself. The current setup, reliant on RapidOCR 3.8.4 and the PaddleOCRv5 `eslav/mobile` recognition model, has proven suboptimal, yielding output that is approximately 70% correct but marred by significant homoglyph substitutions (e.g., 'Хун→Хyн') and other fundamental misrecognitions (e.g., 'Гунфу→Фунфу') . These errors degrade the utility of the extracted text for both user-facing display and downstream enrichment processes. With the recent relaxation of the 4 GB RAM constraint, the primary lever for improvement is the adoption of more powerful, server-grade models that were previously inaccessible due to resource limitations. This section will analyze the strategic upgrade path within the PaddleOCR ecosystem, focusing on the selection of a superior recognition model, the optimization of detection resolution, and the implementation of a robust preprocessing pipeline tailored for the unique challenges of reading entire product packaging.

The most direct and impactful enhancement involves replacing the current `PP-OCRv5 eslav/mobile` recognition model with its `server` counterpart, `PP-OCRv5_server_rec`. While documentation indicates that specialized East-Slavic models may be available in mobile versions, the standard server model is explicitly designed for high-precision tasks and supports a wide range of scripts, including Cyrillic and Latin [[111,183]]. The distinction between mobile and server models is critical; server models are engineered for performance-intensive server deployments where maximum accuracy is paramount, whereas mobile models prioritize computational efficiency for deployment on resource-constrained edge devices [[110]]. Given the new ability to scale the underlying virtual machine (VM), the focus shifts decisively towards accuracy. The OmniDocBench benchmark provides strong evidence for this choice, demonstrating that PP-OCRv5 achieves state-of-the-art results, often outperforming more generalized Vision-Language Models (VLMs) [[46,47]]. Furthermore, subsequent iterations have continued to refine this performance, with PP-OCRv6 showing a +5.1% improvement in recognition accuracy over its predecessor, PP-OCRv5 [[42]]. Even the mobile version of PP-OCRv5 exhibits impressive speed on CPU hardware, processing over 370 characters per second on an Intel Xeon Gold 6271C CPU [[193]], suggesting that the server model, while more computationally demanding, will still operate within acceptable latency bounds for a single-user application. The Apache 2.0 license under which PaddleOCR is released ensures no licensing hurdles for this upgrade [[97]].

Beyond model selection, the detection stage's resolution plays a crucial role in capturing fine details present in printed text, especially on crowded labels. The current configuration uses `limit_side_len=960`, a value chosen primarily to manage memory consumption on a constrained 4 GB VM . Raising this parameter allows the detector to identify smaller text blocks that might otherwise be missed. Research consistently confirms a positive correlation between image resolution and OCR performance [[33]]. Best practices for document scanning recommend resolutions between 300 and 600 dpi for color documents to ensure high-quality transcription [[11,13,35]]. The trade-off between detection accuracy and computational load is a well-understood challenge in OCR pipelines, and increasing the detection limit directly addresses this balance [[32]]. A prudent, incremental approach is recommended. Starting with `limit_side_len=1280` offers a meaningful increase in detail capture without immediately imposing a prohibitive load. If further gains are necessary, this value can be cautiously increased to 1536. This controlled scaling avoids the regression observed in a previous attempt (#114) where unconditional upscaling proved problematic, as this is a deliberate, engine-integrated parameter rather than a naive pre-processing step.

To maximize the effectiveness of the upgraded OCR engine, a standardized preprocessing pipeline is essential. For full-text paragraphs covering an entire package, a consistent set of transformations can significantly enhance input quality before it reaches the detection and recognition stages. Unlike the targeted extraction of a product name, which requires careful cropping and alignment, processing a whole image is less sensitive to minor variations and benefits from robust normalization techniques. A comprehensive OpenCV-based pipeline would include several key steps. First, converting the image to grayscale reduces complexity and focuses the algorithm on luminance information. Second, applying Contrast Limited Adaptive Histogram Equalization (CLAHE) enhances local contrast, which is particularly useful for text that may appear faded, smudged, or unevenly lit against a complex background [[37]]. Third, an adaptive binarization method, such as Otsu's thresholding, creates a clean, high-contrast black-and-white mask that separates the text from the background, making it resilient to varying packaging colors. Finally, a deskew algorithm should be applied to correct for any camera tilt during the photograph, ensuring the text lines are horizontal and aligned. The final image should then be scaled to match the `limit_side_len` used by the detection model (e.g., 1280). This sequence of operations is a standard practice in document analysis software and is designed to create an ideal input for the OCR engine, thereby improving overall fidelity [[37]]. Other libraries like Pillow could also be used for these manipulations, offering similar functionality.

## VM Infrastructure and Cost Analysis for Enhanced OCR Processing

With the decision to adopt a more powerful, server-grade OCR model, the next critical step is to provision a suitable Virtual Machine (VM) instance on Yandex Cloud. The previous 4 GB RAM / 2 vCPU ceiling was a hard constraint that limited the choice of models and configurations, forcing compromises that resulted in lower accuracy . Now, with that constraint relaxed, the objective is to select a VM size that comfortably accommodates the new OCR stack, ensuring low-latency performance for a single-user workflow while remaining cost-conscious. This section provides a detailed analysis of the appropriate VM sizing, justifying the selection based on the resource demands of modern OCR systems, and concludes with a realistic cost estimation using publicly available pricing data.

The resource requirements for an OCR pipeline are primarily driven by the recognition model, which is typically the most memory-intensive component. Server-grade models like `PP-OCRv5_server_rec` are larger and more complex than their mobile counterparts, requiring more RAM for loading the model weights and for intermediate computations during inference [[110]]. Additionally, the detection stage and the overall image processing pipeline consume system resources. To ensure smooth operation, a headroom of at least 4-8 GB of free RAM should be maintained after loading the application and its dependencies. Given that the existing backend services (Caddy, Spring, Postgres) already consume approximately 3.4 GB of the 4 GB VM, expanding beyond this limit is necessary . A mid-range server configuration appears to be the optimal starting point. A VM with **4 vCPUs and 16 GB of RAM** strikes a balance between performance and cost-effectiveness. Four vCPUs provide sufficient parallel processing power to handle both the CPU-bound detection and recognition stages concurrently, minimizing end-to-end latency. Sixteen gigabytes of RAM provides ample space for the large recognition model, the active image buffers, and the operating system, preventing performance degradation due to swapping.

This recommendation aligns with general cloud infrastructure best practices for deploying modern AI workloads. Comparisons of various cloud providers show that instances with 4 vCPU and 16 GB RAM are common and competitively priced. For example, a Linode G7 4x2 instance with an AMD Milan processor, 4 vCPU, and 8 GB of RAM is priced at $43 per month [[113]]. Another example is a DigitalOcean droplet with 4GB RAM costing $24/month, while an Akamai (Linode) 4/80 SSD instance costs $43/month [[113]]. Extrapolating from these figures, a comparable Yandex Cloud instance would fall into a similar price bracket. The specific platform or CPU generation within Yandex Cloud can affect ONNX latency, so it is advisable to choose a modern platform if available [[53,130]]. For instance, choosing a platform with newer-generation CPUs can yield better inference speeds for the OCR models [[130]].

Based on Yandex Cloud's public pricing policies, the cost is calculated based on allocated computing resources (vCPU and RAM), with usage billed in one-minute increments [[50,56]]. The minimum billing unit is one minute, meaning partial minutes of usage are rounded up [[56]]. Attached disks and network egress are charged separately, which is a benefit for this project as photo egress is forbidden, and disk I/O for processing a single image is minimal [[50]].

## Designing a Description-Safe Post-OCR Correction Mechanism

While upgrading the OCR engine improves the base level of accuracy, the resulting text will inevitably contain errors. The true challenge lies in post-processing this output to create a version that is both human-readable and safe for downstream use. The failure of the current `confusable_normalize` function, which turns the word `Букет` into the plausible-sounding but incorrect `Вукет`, serves as a critical cautionary tale . This behavior, while harmless in a catalog-matching context where fuzzy logic can absorb the error, is detrimental for a free-text description intended for human review. The goal of the correction mechanism must be to avoid inventing wrong words, even if it means leaving some obvious garble in place. This section details the design of a "description-safe" correction method, evaluating dictionary-gated approaches as the primary strategy and lightweight seq2seq models as a potential secondary option.

The problem extends beyond simple homoglyph substitution. The error corpus includes genuine misrecognitions, such as `Гунфу→Фунфу` and `северо→соверо`, which a homoglyph mapping alone cannot fix . Therefore, any effective correction strategy must be able to address multiple error types. One promising paradigm is to treat OCR correction as a machine translation task, where the noisy OCR output is translated into clean text [[21,103]]. However, for a local, CPU-only implementation, the complexity of such an approach must be carefully weighed against its benefits. The most practical and reliable initial strategy is a dictionary-gated correction method, which combines conservative homoglyph normalization with morphological validation.

The proposed multi-stage correction process is as follows:
1.  **Conservative Homoglyph Normalization:** The first step is to apply a mapping that converts common Latin and digit homoglyphs back to their closest Cyrillic equivalents. This targets errors like `Byкeт` from `Букет`. The key difference from the flawed `confusable_normalize` is that this mapping is not guaranteed to succeed. It is a best-effort conversion.
2.  **Dictionary-Gated Validation:** After normalization, each resulting token is checked against a comprehensive Russian dictionary. If the normalized token is a valid Russian word, the correction is accepted. If it is not a valid word, the original, uncorrected token from the OCR output is retained. This safety gate directly solves the `Букет→Вукет` problem. In that case, `Byкeт` would be normalized to `Букет`, which is a valid word and is accepted. However, if the OCR had produced `Xyкeт` (from a misread `Х`), it would normalize to `Хукет`. Since `Хукет` is not a real Russian word, the correction would be rejected, and the clearer `Xyкeт` would be left for the user to see, which is far preferable to the plausible falsehood `Хукет`.
3.  **Spellchecking for Misrecognition Errors:** For tokens that pass the dictionary check or remain uncorrected, a spellchecker can be applied to catch misspellings arising from genuine misrecognitions (e.g., `Гунфу`). Libraries like `pyspellchecker` or `hunspell` can suggest corrections for misspelled words, which can then be offered to the user or automatically applied if confidence is high.

For implementation, using `hunspell` with a `ru_RU` dictionary is a highly recommended approach. Hunspell is a mature, open-source spell-checking library with extensive language support. The `ru_RU` dictionary is readily available and can be easily integrated into a Python application via bindings like `hunspell.py` or `pyphen`. This approach is lightweight, fast, and requires minimal additional RAM, making it perfectly suited for a local, CPU-only environment. The dictionary files themselves are small and can be bundled with the application. This strategy prioritizes faithfulness and readability over absolute perfection, which is the correct priority for this use case.

As an alternative, a lightweight sequence-to-sequence (seq2seq) model could be explored. Architectures like T5-small or byT5-small have been successfully adapted for OCR post-correction tasks [[21,27]]. These models can learn complex, context-aware correction patterns that go beyond simple dictionary lookups. However, they come with significant drawbacks. They require more RAM to load the model weights and will introduce higher latency for inference on a CPU compared to a dictionary lookup. Their performance is also dependent on the quality and relevance of the training data, which would need to be generated. Given the success of dictionary-based methods on similar tasks and the resource constraints of a local deployment, the added complexity and overhead of a seq2seq model do not appear justified for the initial implementation [[22,28]]. The dictionary-gated method should be considered the primary and most cost-effective solution.

The following table compares the two main correction strategies.

| Feature | Dictionary-Gated Correction | Lightweight Seq2Seq Model |
|---|---|---|
| **Core Technology** | Hunspell dictionary lookup, regex, and rule-based normalization. | Quantized T5/byT5 model for sequence-to-sequence prediction. |
| **RU Language Support** | Excellent, via standard `ru_RU` dictionaries. | Good, depends on fine-tuning on Russian text. |
| **RAM Usage (CPU)** | Low. Only requires loading the dictionary into memory. | Moderate to High. Requires loading a neural network model (~100-200MB+). |
| **Latency (CPU)** | Very Low. Milliseconds per word/token. | Moderate. Several hundred milliseconds to seconds for a full paragraph. |
| **Safety (Plausible Errors)** | **High.** Rejects invalid words, preserving original OCR text. Directly mitigates the `Букет->Вукет` problem. | **Medium.** Can generalize but may still produce plausible-sounding errors if trained on insufficient or noisy data. |
| **Complexity of Implementation** | Low. Relies on mature, off-the-shelf libraries. | High. Requires model selection, potentially custom training/fine-tuning, and inference scripting. |
| **Primary Use Case Fit** | **Excellent.** Fast, safe, and effective for common homoglyph and spelling errors. | **Good.** Potentially better for complex, context-dependent errors, but overkill for baseline needs. |

In conclusion, the dictionary-gated correction method is the recommended approach. It is fast, lightweight, and inherently safer than blind homoglyph mapping. It directly addresses the primary failure mode identified by the user and provides a significant improvement in the quality and trustworthiness of the final output text. This method will transform garbled text like `Byкeт` into the correct `Букет` while safely rejecting dangerous transformations like `Букет` into `Вукет`.

## Leveraging Noisy OCR Output as Enrichment Context

Once a clean(ish) text string has been generated through the combined efforts of an upgraded OCR engine and a safe post-correction module, the final piece of the puzzle is to leverage this information effectively. The ultimate purpose of extracting the description is not merely to display it to the user, but to feed it into the existing enrichment pipeline to identify and gather information about a new tea that is not yet in the catalog . This requires a strategy for transforming the unstructured, albeit improved, text into structured, actionable context. This section outlines a methodology for parsing the description, extracting key signals, and using them to query the established Wikidata and YandexGPT enrichment tiers in a robust manner.

The primary challenge is to extract discrete, meaningful entities and attributes from the free-text description. Instead of passing the entire paragraph as a monolithic block to the enrichment services, a structured extraction process should be implemented. This allows for more targeted and resilient queries. The extraction targets should focus on the most salient pieces of information found on tea packaging. These include:
*   **Tea Type:** Keywords indicating the broad category of tea, such as `черный чай` (black tea), `зеленый чай` (green tea), `белый чай` (white tea), `уйву` (oolong), `пуэр`, or `жемчужина чая` (pearl tea).
*   **Origin and Region:** Geographical names that indicate where the tea was grown. This could be a country (`Китай`, `Индия`, `Ямал`), a province or region (`Дагестан`, `Алтай`), or even a specific estate or garden name.
*   **Vendor and Brand Name:** The company or brand responsible for the product. This is often a mix of Cyrillic and Latin characters and may include URLs or other identifiers.
*   **Flavor Notes and Sensory Descriptors:** Words that describe the taste and aroma, such as `вкус` (taste), `аромат` (aroma), `пряный` (spicy), `цветочный` (floral), `древесный` (woody), `сладкий` (sweet), `цитрусовый` (citrus), etc.
*   **General Keywords:** Any other descriptive terms that can help characterize the tea, such as brewing instructions (`заваривать`, `температура`).

This extraction can be efficiently performed using a combination of regular expressions and keyword matching against curated lists. For example, a regex pattern can be used to find units like `г` or `мл`, helping to parse nutritional information if present. A list of known tea-growing regions and types can be compiled and matched against the text. This process is computationally inexpensive and runs entirely on the local side before the data is sent to the enrichment tiers. The output is a structured JSON object containing arrays of detected entities for each category.

With this structured data, the queries to the enrichment services can be made far more robust and informative. For the Wikidata tier, the extracted brand names, origins, and tea types can be used to perform entity linking. Wikidata is a knowledge graph, and querying it with specific attributes is more effective than a simple text search. The system can construct SPARQL queries or use Wikidata's API to find items that match the provided criteria. For example, it could search for a `Product` with an `instance of` property pointing to a `tea` entity, and whose `producer` matches the extracted brand name. The fuzzy matching capabilities inherent in the existing enrichment logic will help absorb minor OCR errors in these identifiers .

For the YandexGPT LLM enrichment tier, the structured data can be used to craft a highly effective natural language prompt. Rather than feeding the raw, noisy OCR text, the system can generate a concise summary of the tea's properties. For example: "Identify a tea based on these attributes: origin: 'Ямал', type: 'черный', flavor notes: 'пряный, древесный'." This structured prompting technique is a well-established method for improving the reliability and relevance of LLM responses [[134]]. It directs the model's attention to the most important features and reduces the chance of it being misled by irrelevant or corrupted text. The combination of structured signals from the corrected OCR output provides a rich, multi-faceted context that significantly increases the likelihood of correctly identifying the unknown tea.

Finally, the system must gracefully handle cases where the enrichment process yields uncertain or ambiguous results. It is unrealistic to expect perfect identification every time. The UI should be designed to present these possibilities to the user. For instance, if Wikidata returns multiple potential matches, or if the YandexGPT response expresses low confidence, the application should display these options in an editable field. The user can then review the suggestions, select the correct one, or manually enter a name if none are satisfactory. This completes the feedback loop, placing the user back in control and ensuring that the final entry in the catalog is accurate. This human-in-the-loop approach is essential for a system that relies on imperfect local OCR. By structuring the context intelligently, the system maximizes the signal from the OCR output, enabling the downstream enrichment tiers to perform at their peak potential.

## Integrated Implementation Roadmap and Strategic Recommendation

This research report has systematically deconstructed the problem of extracting clean, full-text Russian tea descriptions and proposed a multi-pronged solution centered on an upgraded local OCR pipeline, a safe post-correction mechanism, and an intelligent enrichment-context strategy. The analysis of Track A confirms that upgrading to a server-grade OCR model is the most direct path to higher fidelity. Track B establishes that a dictionary-gated correction method is the safest and most practical way to fix errors without inventing plausible falsehoods. Finally, Track C details how to structure the resulting text to maximize its utility for downstream enrichment. This concluding section synthesizes these findings into a decisive, actionable implementation roadmap and provides a clear recommendation for moving forward.

The recommended course of action is a phased implementation that delivers immediate, tangible improvements while laying a foundation for future enhancements. The plan prioritizes changes that offer the highest return on investment in terms of accuracy and user experience.

**Phase 1: Core System Upgrade (Weeks 1-2)**
1.  **Upgrade OCR Stack:** Replace the current `PP-OCRv5 eslav/mobile` recognition model with the `PP-OCRv5_server_rec` model. Pin the exact model file from the official PaddleOCR repository (e.g., `ch_PP-OCRv5_server_v2.0_det_infer.onnx` and `ch_PP-OCRv5_server_v2.0_rec_infer.onnx`) [[108]]. Concurrently, increase the detection model's `limit_side_len` parameter from 960 to **1280** to improve the recognition of smaller text elements common on packaging.
2.  **Scale VM Infrastructure:** Provision a new Yandex Cloud VM with the specified **4 vCPU and 16 GB RAM** configuration. Based on typical pricing, the estimated monthly cost is approximately **₽8,500 - ₽12,000 RUB** (as a placeholder; live Yandex Cloud pricing should be verified). This provides the necessary resources to run the server model efficiently without latency issues.
3.  **Implement Preprocessing Pipeline:** Integrate a standardized preprocessing pipeline into the OCR sidecar. This pipeline should consist of: grayscale conversion, CLAHE for contrast enhancement, adaptive binarization, deskewing, and final scaling to 1280 pixels on the longest side. This will standardize inputs and improve OCR accuracy.
4.  **Deploy Dictionary-Gated Correction:** Develop and integrate the "description-safe" correction module. This will use the `hunspell` library with a `ru_RU` dictionary to perform conservative homoglyph normalization followed by morphological validation. This directly mitigates the critical failure mode of creating plausible-but-wrong words.

**Phase 2: Enrichment Integration and Refinement (Weeks 3-4)**
1.  **Develop Context Extraction Module:** Create a parser that takes the corrected OCR text and extracts structured signals: tea type, origin, vendor, and flavor notes, using curated keyword lists and regular expressions.
2.  **Integrate with Enrichment Tiers:** Modify the existing enrichment logic to consume the structured data from Phase 2. Construct targeted queries for the Wikidata API and generate structured prompts for the YandexGPT LLM. Implement a UI flow to present uncertain enrichment results for user review.

Upon completion of this roadmap, the TeaTiers application will undergo a significant qualitative improvement. The user-facing description field will no longer be a frustrating source of garbled text. Instead, it will be a clean, readable, and largely accurate representation of the package's description, requiring only light manual edits. This dramatically lowers the barrier to entry for adding a new tea to the catalog. More importantly, the structured context generated by the system will provide the downstream enrichment tiers with high-quality signals, leading to more accurate and reliable identification of unknown teas. The gap between the current state (where users may feel compelled to retype the entire description) and the desired state will be substantially narrowed, shifting the user's effort from correction to confirmation.

**Strategic Go Decision:**
The proposed local-first approach is not only sufficient but is the strategically correct path forward. It fully satisfies the core constraints of the project: it operates entirely on a user-controlled VM, it prevents photo egress, and it maintains the GMS-free, no-PII architecture . The investment in a slightly larger VM is a worthwhile expenditure to achieve the primary goal of high-fidelity local OCR. There is no compelling reason to introduce an opt-in cloud-based OCR path for the extraction step. Such a path would compromise the privacy and autonomy principles central to the TeaTiers project and would add unnecessary complexity and dependency. The combination of a powerful server-grade OCR model, a safe and intelligent correction layer, and a smart context-generation strategy provides a robust, private, and highly effective solution to the stated research goal. We should proceed with the implementation plan outlined above.
