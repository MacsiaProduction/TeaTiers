# Сравнение вариантов

| Критерий                | PostgreSQL `pg_trgm`        | Meilisearch CE                          | Typesense                           | OpenSearch (Elasticsearch)    |
|-------------------------|-----------------------------|-----------------------------------------|-------------------------------------|-------------------------------|
| **Точность поиска (typo-толерантность)** | *Позволяет полнотекстовый поиск с опечатками*. Поддерживает поиск по подобию: `WHERE name % 'запрос'` (символ `%` – оператор pg_trgm) или `ILIKE '%запрос%'`. Результаты можно сортировать по похожести: `ORDER BY similarity(name,'запрос') DESC`. Для топ-N-поиска возможен KNN-запрос с GiST: `ORDER BY name <-> 'запрос' LIMIT N`. Работает хорошо для кириллицы и латиницы (регистр игнорируется по умолчанию). Тоны в пиньине можно игнорировать с помощью `unaccent` (удаляет диакритические знаки). | *Очень высокая точность при опечатках*. Из коробки поддерживает дамерау–левенштейновы ошибки (swap, инсерция, удаление символов) и выдаёт релевантные совпадения. Быстрое ранжирование. Однако поддержку китайского текста (и пиньиня) нужно проверять: Meili разбивает текст по символам (без явной сегментации слов), так что одиночный символ-опечатка может не находиться. | *Высокая точность и адаптивность*. Поддерживает локализацию (ICU) и “умные” токенизаторы для разных языков. Для `locale: "ru"` или `"en"` автоматически убирает диакритики и учёт регистра по ICU. Для китайского (`locale: "zh"`) реализована *символьная сегментация и нормализация вариантов*. Поддерживает тоновые метки пиньиня аналогично другим диакритикам. Имеет гибкие настройки (параметр `typoTolerance`, `synonyms` и др.). | *Мощная и гибкая полнотекстовая система (Lucene)*. Позволяет очень тонко настраивать анализ текста: есть плагины для китайского (Smart Chinese, IK-инсталляция), для других языков. Поддерживает fuzzy-поиск (запрос вида `fuzziness=1` даст подбор по опечаткам) и гибкое ранжирование (BM25 и настраиваемые функции). Требует явной настройки tokenizer/analyzer для разных языков. |
| **Поддержка CJK (китайский)** | Ограничена. `pg_trgm` строит триграммы по байтам: китайские иероглифы многобайтовые, и по умолчанию делятся “по 3 байта”, а не по символам. Для полноценной работы нужно, чтобы локаль БД была UTF-8 (не `C`) – тогда pg_trgm будет разбивать китайский по символам. Но качество снижается: Tom Lane предупреждает, что “`pg_trgm` практически бесполезен для текста, состоящего главным образом из многобайтовых символов”. Подстрока ‘龙井茶’ *(%LIKE%)* может давать полезные совпадения, но мелкие ошибки (например, одна неверная иероглифа) могут не угадываться. Короткие CJK-запросы (<3 символов) pg_trgm вообще не обрабатывает эффективно. | **Нативно поддерживает китайский** с подсегментацией (наивная сегментация символами). Типесеренс при локали `zh` делит текст на символы и учитывает варианты (упрощённые/традиционные). Ошибки на уровне отдельных иероглифов подбирает лучше. *MeiliSearch* не имеет специальной фонетической сегментации для китайского: поисковые запросы токенизируются построчно без ML-токенизатора, так что одиночные иероглифы-ошибки трудно ловить. | **Поддерживает китайский через специальные анализаторы**. Имеются плагины `analysis-smartcn`, `analysis-ik`, `jieba` и др. (их нужно установить/включить). С их помощью можно добиться сегментации по словам и коррекции опечаток. В целом OpenSearch может быть настроен для CJK, но это сложнее. |
| **Лёгкость эксплуатации и требования** | **Очень просто** (встроенная БД). Дополнительный сервис не нужен: поиск по колонке `name` реализуется SQL/GIN-индексом. Использует GIN-индекс `gin_trgm_ops` (в примере проект уже имеет `tea_name_trgm_idx`). Потребляет память и CPU по объёму данных в таблице; при 50k записей это незначительно. Нет отдельного процесса (нет RTE). Поддерживается `postgres:16-alpine` (pg_trgm и unaccent – стандартные расширения). Минус – поиск ограничен возможностями SQL. | **Средняя сложность**. Нужно запустить отдельный контейнер `meilisearch`. Требует ~сотню мегабайт ОЗУ минимум (LMDB подтягивает набор в память; на 50k документов – вероятно <300 MB). Управление – простое: REST API. Необходима синхронизация: нужно писать логику занесения данных через API. Java-клиент (Kotlin) доступен. Позволяет быстро настроить поля и ранжирование через API/панель. | **Относительно просто**. Отдельный контейнер `typesense-server` (Rust) – стартует ~20 MB пустой, под данные ~2-3×T резерва RAM (T=объём текстовых полей). Удобная документация. Есть официальный Java-клиент, можно интегрировать через REST. Настройка – через HTTP, конфигурация коллекций с полями и опциями. Синхронизация – аналогична Meili (см. ниже). Требует немного больше ручных настроек, чем Meili, но меньше, чем OpenSearch. | **Сложнее**. Требует JVM (~512 MB ядра) и ОС-поддержки. Docker-образ (OpenSearch 3.x) обычно запускают с ≈2–4 GB heap. Настройка сложнее: нужно создавать индекс, определять анализаторы и маппинги. Интеграция возможна через Spring Data Elasticsearch или OpenSearch Java client. Управление – тяжелее (JVM GC). На 50k записей – система раздута, но медиа CPU- и RAM-запросы останутся низки; тем не менее низкоуровневые операции (индексация, merge) тяжелее. |
| **Синхронизация из Postgres** | **Не требуется** – поиск по той же БД. После записи в таблицу данные сразу участвуют в поиске. Единственная доработка – создание выражений-индексов: например, дополнительный столбец `name_unaccent = unaccent(lower(name))` с GIN-индексом для пиньиня. | **Требует свою логику**. Придётся при изменении данных делать HTTP-запросы в Meili API. Варианты: простейший – периодический reindex всего каталога; лучше – по полю `updated_at` отправлять только новые/обновлённые записи через Bulk API; также можно реагировать на события (триггеры, Debezium). Meili не предоставляет готового коннектора к Postgres, но общая схема – аналогична Typesense. | **Гибкие стратегии**. Typesense даёт рекомендации: например, *пуллинг* по полю `updated_at` и Bulk-API upsert; *триггеры/слушатели* с быстрым пушем в очередь; или даже Sequin (репликация). Существуют инструменты (например, Sequin, Airbyte) для автоматической репликации Postgres→Typesense. Java-клиент легко вставляет данные. | **Нужны коннекторы или ручной код**. Можно писать в OpenSearch через REST или JDBC River (устарело). Обычно делают приложение, которое при изменении в Postgres шлёт данные в OpenSearch (Bulk API). Есть инструменты CDC (Debezium+Kafka→OpenSearch Sink). Более трудоёмко, чем с Meili/Typesense. |
| **Лицензия (безопасность)** | Полностью открыт, входит в PostgreSQL (Postgres – лицензия PostgreSQL/МIT-like). Расширения `pg_trgm`, `unaccent` – тоже стандартные (GPL/PGLv2 compatible). | Комьюнити-издание – **MIT**, можно использовать в приложении без ограничений. | **Apache 2.0** (разработчики упоминают лицензию Apache 2.0, см. исходники). Открытый код, без «ловушек». | **Apache 2.0** (fork Elasticsearch от AWS, полностью открытый с разрешающей лицензией). |

# Рекомендуемый MVP

**Подход:** использовать PostgreSQL + расширение `pg_trgm` в качестве поискового индекса, с доработкой на стороне БД для мультиязычности и упрощения. Причины: не нужно заводить новый сервис, достаточно малый каталог, все данные уже в БД.  

**Настройка:**

- **Индекс:** уже есть `tea_name_trgm_idx ON gin(name gin_trgm_ops)` (gin_trgm_ops). Убедиться, что база в UTF-8 и локаль не `C` (например, `LC_CTYPE='ru_RU.utf8'`), чтобы правильно разбивались русские и китайские символы.
- **Запросы:** использовать запросы вида  
  ```sql
  SELECT t.id, tn.name, similarity(tn.name, :query) AS sim
  FROM tea t
  JOIN tea_name tn ON t.id = tn.tea_id
  WHERE tn.locale = :loc AND tn.name % :query
  ORDER BY sim DESC;
  ```  
  Здесь оператор `%` – индексный оператор pg_trgm для похожести. Порог `pg_trgm.similarity_threshold` (по умолчанию 0.3) можно понизить (например до 0.2) для более «широкого» поиска.
- **Ранжирование:** использовать `ORDER BY similarity(name, :query) DESC`. Либо если нужна жёсткая топ-N, создать GiST-индекс:  
  ```sql
  CREATE INDEX tea_name_trgm_gist_idx ON tea_name USING gist (name gist_trgm_ops);
  ```  
  и выдавать `ORDER BY name <-> :query LIMIT 10`.
- **Обработка языков:**
  - *Русский, английский:* обычный `ILIKE '%…%'` или `%` работает без изменений.
  - *Пиньинь:* включить расширение `unaccent` (если не включено): `CREATE EXTENSION unaccent;`. Можно создать функциональный индекс:  
    ```sql
    CREATE EXTENSION pg_trgm;  -- уже есть
    CREATE EXTENSION unaccent; -- убирает тону
    CREATE OR REPLACE FUNCTION f_unaccent(text) RETURNS text
      AS 'SELECT unaccent(\'unaccent\', $1)' LANGUAGE SQL IMMUTABLE;
    CREATE INDEX tea_name_unaccent_trgm ON tea_name USING gin (f_unaccent(name) gin_trgm_ops);
    ```  
    Тогда в запросе можно делать `WHERE f_unaccent(name) % f_unaccent(:query)` для поиска без учета тоновых знаков. (Или добавить колонку `name_unaccent` и индексировать её.)
  - *Китайский:* добавить условие `locale = 'zh-Hans'` для китайских названий. Если поиск по китайским слогам (пиньинь), аналогично использовать `unaccent` и нижний регистр. По иероглифам: просто искать `ILIKE %символ%` и `%`-оператор pg_trgm; однако помним, что качество невысокое (см. ограничения ниже). Можно рассмотреть наложение условий (например, перебор по символам).
- **Обработка коротких запросов (<3 символа):** `pg_trgm` не индексирует результаты для очень коротких запросов. Для таких случаев fallback – обычное `LIKE 'q%'` (с префиксом) без индекса или полное сканирование, так как данных мало. Можно создать дополнительный материализованный помощник (например, таблицу слов или таблицу символов из названий) как описано в доке, но для 50k записей это, вероятно, избыточно.

**Пример SQL:**  
```sql
SET pg_trgm.similarity_threshold = 0.2;
SELECT t.id, tn.name, similarity(tn.name, 'маття') AS sim
FROM tea t JOIN tea_name tn ON t.id = tn.tea_id
WHERE tn.locale = 'ru'
  AND f_unaccent(tn.name) % f_unaccent('маття')
ORDER BY sim DESC;
```  
Здесь `f_unaccent` – пример функции для удаления тонов/акцентов.  

# Резервный вариант

Если pg_trgm/SQL-поиск не даёт достаточного качества (особенно по китайскому, коротким запросам или большим ошибкам), резерв: внедрить специализированный поисковый движок. Наиболее подходящий – **Typesense**. 

- Лицензия Apache 2.0 (без ограничений).  
- Поддерживает кириллицу, латиницу и китайский (локаль `zh` с сегментацией).  
- Точность поиска по опечаткам выше, чем у чистого LIKE.  
- Kotlin/Java-клиент есть.  
- Процесс: поднять контейнер Typesense (потребляет десятки Мб пустым, и далее ~2-3× размера данных в RAM), создать коллекцию `teas` с полями `locale` и `name`, **синхронизировать** данные из Postgres (см. ниже).  
- API позволяет сложный поиск: задать мультиполе, `filter_by=locale:..., query_by=name`, включить `typoTolerance` и др.  

В альтернативу (менее предпочтительно из-за памяти и сложности) можно рассмотреть OpenSearch, но это «тяжёлый» вариант для маленького приложения и требует большой настройки.

# План оценки (золотой набор запросов)

Чтобы проверить качество поиска, составим ~20-30 тестовых запросов (с опечатками) и ожидаемое поведение (примеры названы условно):

- Русский: 
  - Запрос «чай» → должна найти «чай», «чайот» и подобные. Пример «чай» vs «цаи» (опечатка). 
  - «черный» vs «чёрный» (с акцентом) → игнорируется разница (unaccent/pg_trgm не чувствителен).
  - «маття» vs «матча» (опечатка «тт»→«ч»). 
  - «пуэр» vs «пуэр» (правильно).
  - «пуэр» vs «пра» (несколько ошибок) – должен найти «пуэр», но с низкой симметрией.
- Английский:
  - «green tea» vs «gren tea» (перестановка «r»).
  - «oolong» vs «olong» (удалена буква).
  - «oolong tea» vs «oolng ta» (2 опечатки).
  - «chai latte» vs «hai latte» (пропущена буква).
  - «lap-sang souchong» vs «lapsang souchng» (пропуск «o»).
- Пиньинь (латинские буквы с тонами/без):
  - «běijīng» vs «beijing» (снечтение тонов).
  - «changwen» vs «chāngwén» (ответить на тон).
  - «luan guapian» vs «lu'an guapian» (апостроф).
  - «luan guapian» vs «luanguapian» (без пробела).
- Китайские иероглифы:
  - «龙井茶» (Лунцзин ча) точно.
  - «龙井 茶» (с пробелом) – тоже.
  - «泷井茶» (замена «лун» на похожий «лунь») – должен найти Лунцзин, возможно низкий приоритет.
  - «普洱茶» (пуэр) точно.
  - «普爾茶» (традиционное написание) – если есть (если нет, ожидаем ~похоже по Unicode).
- Мульти-языковые: «black longjing» (английская ошибка + китайский) – ожидаем найти «龙井茶» и «Black Dragonwell» (если есть аналог).
- Короткий запрос: «茶» – искать все напитки с «茶». (`pg_trgm` без wildcard обычно не сработает, но LIKE найдет).
- Короткий латинский: «olong» – найти «oolong».

В этих примерах проверяется: толерантность к смене букв, отсутствию диакритики, смешиванию языков, пробелам/апострофам.  

# Чего **не делать**

- **Не полагаться на «волшебные» библиотеки распознавания/векторизации** для схожести, если задача – простые опечатки. Это избыточно (нет задачи семантики, только строковый поиск).  
- Не пытаться всякий раз подгружать внешние данные (веб-краулинг, словари) – система работает с локальным каталогом.  
- Не использовать коммерческие решения (Firebase, Algolia, Google) – требования “без GMS” и “открытый софт”.  
- Не задавать слишком низкий threshold (слишком свободные совпадения) – иначе нерелевант попадёт.  
- Не дублировать лишние сервисы без необходимости (например, не вводить ElasticSearch для 50k строк, если PostgreSQL справляется).  
- Не забывать проверять реальные метрики памяти/производительности перед деплоем: например, оценить объем RAM у Meili/Typesense с нашим набором или доступность нужных расширений (pg_trgm, ICU) в целевом Docker-образе. 

**Источники:** официальные доки PostgreSQL по `pg_trgm` и `unaccent`; Tom Lane о CJK-ограничениях; руководство Typesense по локалям; лицензии Meilisearch и Typesense; официальные гайды по синхронизации с Typesense.
