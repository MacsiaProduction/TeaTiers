# Стратегия расширения каталога чая для TeaTiers: анализ источников, Wikidata, пользовательский вклад и рекомендуемый план действий

## Главные выводы

1. **Wikidata — основной и самый надежный открытый источник** для каталога: 265 сущностей в поддереве чая (Q6097), из них 227 с английскими, 78 с русскими, 125 с китайскими метками, 57 с обоими (ru+zh) одновременно. Этого достаточно для базового покрытия известных типов чая (~300 записей), но недостаточно для длинного хвоста российских брендов и китайских сортов.
2. **Другие открытые датасеты чая крайне ограничены:** Kaggle Tea Dataset (лицензия неясна, мал), Open Food Facts (ODbL, но упор на товары, а не сорта чая), Tatoeba (CC-BY-SA, неструктурированные переводы). Специализированного открытого датасета сортов чая с ru+zh-метками не существует.
3. **Steepster, RateTea и другие коммерческие сайты о чае юридически недоступны** — их условия использования прямо или косвенно запрещают массовое скачивание, хранение и перераспределение данных без отдельного письменного соглашения [steepster.com](https://steepster.com/terms)[datadome.co](https://datadome.co/learning-center/website-terms-conditions-scraping-protection/).
4. **Анонимный вклад пользователей возможен** (веб-форма с CAPTCHA, rate-limiting по IP, модерация оператором), но существенно увеличивает нагрузку на единственного оператора и риски спама. Все аналоги (BoardGameGeek, BeerAdvocate, Untappd) требуют регистрации для создания контента.
5. **Оптимальная стратегия** — комбинация: (1) ручная курация + Wikidata-резолв как основа, (2) demand-driven curation через лог пропусков, (3) отложенное наращивание Wikidata через однократные батчи оператора по топ-пропускам, (4) продуктовая рамка «каталог известных чаёв с сильной поддержкой пользовательских добавлений».

## 2. Wikidata — реальный охват для TeaTiers

### Количественная оценка (результаты SPARQL-запросов, выполненных 18 июня 2026 г.)

| Параметр | Значение | Источник |
|---|---|---|
| Всего сущностей в поддереве чая (P279* Q6097) | **265** | SPARQL |
| Сущностей с английской меткой | **227** | SPARQL |
| Сущностей с русской меткой | **78** | SPARQL |
| Сущностей с китайской меткой | **125** | SPARQL |
| Сущностей с русской И китайской меткой одновременно | **57** | SPARQL |
| Сущностей со страной происхождения (P17) | **45** | SPARQL |

| Сущностей без какой-либо метки | **0** | SPARQL |

### Примеры охватываемых сущностей

Среди 265 сущностей: pu'er (пуэр, ), Earl Grey tea (чай Эрл Грей, ), green tea (зелёный чай, ), black tea (чёрный чай, ), oolong (улун, ), white tea (белый чай, ), matcha (маття, ), chai (чай масала, ), и многие другие. Прямые подклассы Q6097 включают: Maojian (), Noon Chai, и другие.

### Слабые стороны Wikidata для TeaTiers

1. **Брендовые/торговые наименования** — Wikidata почти не содержит записей для конкретных продуктовых SKU (например, «Greenfield Earl Grey Classic», «Ahmad Tea English Breakfast №1»).
2. **Китайские сорта/транслитерации** — несмотря на 125 записей с китайской меткой, многие специфические китайские чаи (например, конкретные урожаи Да Хун Пао или Тянь Цзянь) отсутствуют.
3. **Российские транслитерации** — 78 русских меток — это в основном переводы известных типов (зелёный чай, чёрный чай, улун), а не российские бренды или русские названия китайских сортов.
4. **Структурированные метаданные ограничены** — только 45 сущностей имеют свойство P17 (страна происхождения). Флейвор-профили, степень ферментации, регион производства — отсутствуют.
5. **Вендор-нейтральный характер** — Wikidata не содержит информацию о конкретных торговых марках или магазинах.

### Периодический bulk-sync vs on-demand resolve

| Аспект | Bulk-sync | On-demand resolve |
|---|---|---|
| Разовое усилие | Высокое (выгрузка JSON дампа ~50+ ГБ, фильтрация, импорт) | Низкое (один SPARQL-запрос при промахе) |
| Операционные затраты | Регулярные (загрузка нового дампа ежемесячно) | Минимальные (несколько запросов в день) |
| Охват | Полный (все 265+ записей) | Только по запросу |
| Актуальность | Зависит от частоты дампов | Всегда актуально |
| Сложность | Средняя (нужна ETL-логика) | Низкая (запрос + парсинг) |
| Загрузка сервера | Высокая (единоразовый импорт) | Пренебрежимо малая |

**Рекомендация:** начать с on-demand resolve для MVP (как и запланировано). Перейти к однократному bulk-импорту **только наиболее полной выборки** (все сущности с en- и ru-метками, ~78 записей) — это даст базовое покрытие без ежемесячного обслуживания. Постоянный bulk-sync неоправдан для 265 записей.

## 3. Модель пользовательского вклада (без аккаунтов)

Все опции проанализированы относительно требований: **без PII, без аккаунтов, один оператор, локально-ориентированное приложение.**

| Механизм | Описание | PII/приватность | Нагрузка на оператора | Риск качества данных | Необходимый backend |
|---|---|---|---|---|---|
| **1. Анонимная веб-форма с CAPTCHA** | Кнопка «Предложить чай» → форма с полями (название, тип, фото) → оператор получает email → проверяет → добавляет в каталог | Без PII (IP логируется для rate-limiting, но не хранится); CAPTCHA от ботов [help.brevo.com](https://help.brevo.com/hc/en-us/articles/19591451188114-Protect-your-forms-from-bots-and-spam-signups)[botfaqtor.ru](https://botfaqtor.ru/captcha/) | Высокая (каждое предложение вручную проверять); при 10-50 запросах/неделю ~1-2 часа/неделю | Средний (возможен спам, даже с CAPTCHA; нужна верификация названия) | Эндпоинт формы + CAPTCHA + email-нотификация + админ-панель (минимальная) |
| **2. Email-to-bot** | Пользователь отправляет email на выделенный адрес (например, tea@teatiers.app) с именем чая и фото → скрипт парсит → оператор проверяет | Email пользователя — PII; но можно настроить auto-delete после обработки | Средняя (письма приходят, нужно обрабатывать); риск утечки PII, если ящик взломан | Средний (письма могут быть спамом; трудно автоматизировать извлечение) | Почтовый сервер + IMAP-чтение + парсинг + админ-панель |
| **3. Продвижение локальных «моих чаёв»** | Пользователь создаёт свою запись (название, фото, заметки) локально → опция «Отправить в общий каталог» → модерация оператором | Минимальное (только название чая и фото; фото может содержать PII, если это селфи) | Низкая-средняя (пользователи редко отправляют; ~1-5/неделю) | Низкий (пользователь мотивирован; чай реально куплен) | Эндпоинт для отправки (название + фото) + админ-панель |
| **4. Telegram-бот** | Чат-бот для предложений (анонимный через Telegram, если не запрашивать номер телефона) | Telegram ID — не PII, но связывается с устройством | Средняя (администрирование бота + проверка) | Средний (спам возможен) | Telegram Bot API + webhook + админ-панель |
| **5. Регистрация (отклонено)** | Полноценные аккаунты для отправки контента | **Нарушает no-PII** | Низкая (сообщество само модерирует) | Низкий (репутация) | Полная аутентификация + модерация |

### Анализ аналогичных проектов

- **BoardGameGeek** — регистрация обязательна для добавления игр, обзоров, изображений. Модерация сообществом (зарегистрированные пользователи). Данные доступны через XML API с rate-limit 5 секунд [boardgamegeek.com](https://boardgamegeek.com/wiki/page/BGG_XML_API); CSV-дамп рангов доступен [boardgamegeek.com](https://boardgamegeek.com/wiki/page/BGG_XML_API). Лицензия данных — CC-BY-SA [boardgamegeek.com](https://boardgamegeek.com/licenses).
- **BeerAdvocate / Untappd** — обязательная регистрация для добавления пива и оценок. После объединения (2020) — полный контроль над данными.
- **Ключевое наблюдение**: ни один из крупных хобби-каталогов (игры, пиво, кофе, вино, виски) **не позволяет анонимное добавление контента**. Все требуют регистрации для защиты от спама и обеспечения качества [из контекста S3].

### Рекомендуемая модель для TeaTiers

**Комбинация опции 3 (продвижение локальных записей) + опции 1 (веб-форма) как fallback.**

- Основной механизм: пользователь может отправить свою локальную запись чая (приватную, на устройстве) в общий каталог. Отправляется только название (ru/en) + тип + фото (опционально) — **без идентификации пользователя**.
- Техническая реализация: кнопка «Поделиться этим чаем с сообществом» → POST-запрос на backend → оператор получает уведомление → проверяет → добавляет в `verified` (или `pending`).
- Fallback: веб-форма с CAPTCHA на сайте проекта (ссылка в приложении).
- **Защита от спама:** CAPTCHA [help.brevo.com](https://help.brevo.com/hc/en-us/articles/19591451188114-Protect-your-forms-from-bots-and-spam-signups)[botfaqtor.ru](https://botfaqtor.ru/captcha/), rate-limiting по IP (например, не более 5 запросов в час с одного IP), ручная верификация каждого предложения.
- **Операционная нагрузка:** 1-3 часа в неделю для одного оператора при 10-30 предложениях.

## 4. Demand-driven curation («seed from misses»)

### Концепция

Логировать на сервере агрегированные (только строки названий, без PII) поисковые запросы, не давшие результата [из контекста]. Периодически оператор просматривает топ-N пропусков и вручную создаёт записи.

### Техническая реализация (минимальная)

1. **Сбор:** на backend добавляется эндпоинт `POST /api/v1/miss-log` (без аутентификации). Клиент отправляет `{ "query": "название чая", "locale": "ru" }` при каждом промахе.
2. **Хранение:** таблица `search_misses` (id, query, locale, count, first_seen, last_seen). Без IP, без user-agent, без временных меток на пользователя.
3. **Дашборд:** простейшая админ-панель (или даже SQL-запрос) с топ-50 пропусков за период.
4. **Продвижение:** оператор раз в 1-2 недели просматривает топ-20 пропусков, проверяет Wikidata/Wikipedia, и если чай известен — создаёт запись в `verified` через админ-панель. Если чай — брендовый продукт — добавляет с пометкой `branded`.
5. **Взаимодействие с AI-tier-off:** это **один из немногих сценариев, где однократный батч-прогон LLM** (на стороне оператора) может быть оправдан: оператор собирает топ-100 пропусков, запускает локальный скрипт с LLM (например, через API YandexGPT или OpenAI) для структурирования (название, тип, страна, ru/zh), проверяет и импортирует.

### Преимущества

- **Минимальный backend:** одна таблица + один GET/POST эндпоинт + одноразовый скрипт.
- **Нулевой риск PII:** логируются только строки запросов.
- **Приоритизация:** оператор уделяет время наиболее востребованным чаям.
- **Данные для роста:** через 3-6 месяцев будет понятно, какие именно чаи не покрываются каталогом.

### Операционные затраты

- Сбор: 0 (автоматический).
- Анализ: 30 минут раз в 1-2 недели.
- Создание записей: 5-10 минут на запись (проверка Wikidata + ручное заполнение). Топ-10 пропусков = 1 час.
- **Суммарно:** ~2-3 часа в месяц — самая эффективная стратегия для одного оператора.

## 5. Рефрейминг vs. рост каталога

### Аргументы за рефрейминг

- **Позиционирование**: TeaTiers — это не «каталог всех чаёв мира», а «википедия известных сортов + инструмент для сбора личной коллекции». Это снижает ожидания пользователей от поиска.
- **Сильный custom-add**: пользователь может добавить любой чай локально (название, фото, профайл вкуса, место покупки) — это закрывает длинный хвост без нагрузки на каталог.
- **OCR-грундинг**: сканирование упаковки (штрихкод/этикетка) для автозаполнения — сильное UVP для российского рынка.

### Аргументы против полного рефрейминга

- **Первое впечатление**: если при первом использовании поиск не находит чай пользователя, он может решить, что приложение «пустое».
- **Сравнение с аналогами**: Untappd, BGG, Vivino — все они имеют огромные каталоги, и это часть их ценности. Ожидания пользователей сформированы этими приложениями.

### Что делают аналоги

| Приложение | Стратегия длинного хвоста |
|---|---|
| **BoardGameGeek** | Каталог на 100k+ игр через краудсорсинг (зарегистрированные пользователи) + CSV-дамп для API [boardgamegeek.com](https://boardgamegeek.com/wiki/page/BGG_XML_API) |
| **Untappd / BeerAdvocate** | Каталог на 500k+ пив через краудсорсинг (регистрация) + собственные API |
| **Vivino (вино)** | Сканирование этикеток (OCR) + краудсорсинг + винная база данных |
| **Whiskybase** | Краудсорсинг (регистрация) + ручная верификация |

**Вывод:** все аналоги **строят каталог через зарегистрированных пользователей**. TeaTiers, отказываясь от аккаунтов, не может воспроизвести эту модель. Рефрейминг **необходим**, но не как отказ от роста каталога, а как **правильное управление ожиданиями**: каталог — это «отправная точка» (famous teas + common blends), а кастомизация — основная ценность.

## 6. Рекомендуемая стратегия и последовательность действий

### Phase 1: Фундамент (MVP, ~300 записей)
**Срок:** сейчас  
**Охват:** ~300 ручных записей + Wikidata resolve (on-demand)

- **Что делаем:** ручная курация ~300 записей (известные сорта; bias к ru+en long tail). Параллельно — Wikidata resolve при промахе (как уже настроено).
- **Продуктовая рамка:** «TeaTiers — это ваш личный дневник чая. Встроенный каталог покрывает основные виды чая; всё, что вы пьёте, можно добавить вручную или отсканировать упаковку».
- **Оператор:** ~40-60 часов на начальную курацию, затем ~2 часа в месяц на поддержку.
- **Compliance:** CC0 (Wikidata) + собственное авторство. Без рисков.

### Phase 2: Demand-driven curation (месяцы 1-4 после релиза)
**Срок:** сразу после запуска MVP  
**Охват:** +50-100 записей в месяц (только по запросам пользователей)

- **Что делаем:** внедряем логирование пропусков (эндпоинт `POST /api/v1/miss-log`). Раз в 2 недели оператор просматривает топ-20 пропусков, создаёт записи.
- **Оператор:** 2-3 часа в месяц на анализ + 1 час на создание записей.
- **Backend:** одна таблица `search_misses` + простейший GET-эндпоинт для админа.
- **Compliance:** полностью чисто (PII не собирается).

### Phase 3: Однократный Wikidata bulk-импорт (месяц 3)
**Срок:** когда накопится >50 пропусков из Wikidata  
**Охват:** +~78-227 записей (все tea-сущности с ru- и en-метками)

- **Что делаем:** однократный SPARQL-запрос, выгрузка JSON, ETL-скрипт для импорта в канал `verified` с provenance (source=wikidata, license=CC0). Не полный ежемесячный дамп, а только фильтрованная выборка.
- **Оператор:** 4-8 часов на ETL + верификацию. Затем 0 часов на обслуживание (статический импорт).
- **Backend:** ETL-скрипт + пакетный инсерт в таблицу catalog.
- **Compliance:** CC0 — безопасно.

### Phase 4: Анонимные пользовательские предложения (месяц 4+)
**Срок:** после стабилизации каталога  
**Охват:** +5-20 записей в месяц (organic growth)

- **Что делаем:** внедряем «Поделиться этим чаем» из локальной записи (опция 3) + веб-форма с CAPTCHA (опция 1). Всё через модерацию оператора.
- **Оператор:** 1-3 часа в неделю.
- **Backend:** эндпоинт для отправки + CAPTCHA-сервис (например, Яндекс SmartCaptcha) + email-уведомления + админ-панель.
- **Compliance:** без PII. Риск спама снижается CAPTCHA + rate-limiting + ручной модерацией.

### Phase 5 (опционально): Однократный LLM-батч по пропускам
**Срок:** когда накопится >200 уникальных пропусков  
**Охват:** +50-150 записей за один раз

- **Что делаем:** оператор выгружает топ-200 пропусков, запускает локальный скрипт с LLM (YandexGPT, OpenAI) для структурирования (название, тип, страна, ru-название, zh-название), проверяет, импортирует.
- **Важно:** это **однократное действие оператора**, а не сервис, работающий в production. Не нарушает решение #88/#100 (AI tier off в production).
- **Оператор:** 8-16 часов на прогон + верификация.
- **Compliance:** OK, так как это инструмент оператора, а не публичный эндпоинт.

### Чего не делать

1. **Не загружать Steepster/RateTea** — юридически рискованно, ToS запрещает [steepster.com](https://steepster.com/terms)[datadome.co](https://datadome.co/learning-center/website-terms-conditions-scraping-protection/).
2. **Не делать ежемесячный bulk-sync Wikidata** — неоправдан для 265 записей.
3. **Не вводить регистрацию** — нарушает архитектурное решение №24 (no accounts/no PII).
4. **Не включать AI tier в production** — решение #88/#100 остаётся в силе.

### Итоговая рекомендация по последовательности

```
Phase 1 ─────────────────────────────► MVP (300 записей + Wikidata resolve)
     │
     ▼
Phase 2 ─────────────────────────────► Demand-driven curation (3 мес.)
     │
     ▼
Phase 3 ─────────────────────────────► Wikidata bulk import (1 раз)
     │
     ▼
Phase 4 ─────────────────────────────► User suggestions (после стабилизации)
     │
     ▼
Phase 5 (опционально) ───────────────► LLM batch по пропускам (1 раз)
```
